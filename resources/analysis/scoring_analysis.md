# EBU6304 评分策略分析

## 评分公式
**个人分 = 小组分 × 个人贡献系数**

---

## 第一次评估（30分，3月22日）

### 小组分拆解

| 得分项 | 满分 | 拿满分的关键 |
|--------|------|-------------|
| 用户和范围定义 | 2 | 清晰列出 TA / MO / Admin 三类用户，范围边界明确 |
| User Story 写法 | 5 | 严格使用 "As a [role], I want [goal], so that [benefit]" 格式，附带 Acceptance Criteria |
| Product Backlog 完整性 | 3 | 覆盖所有角色的核心功能，有优先级(MoSCoW)、估算(Story Points)、迭代分配 |
| 原型功能性 | 5 | 主流程可走通：注册→发职位→申请→审核→结果 |
| 原型易用性 | 5 | 界面清晰，操作直觉，有错误提示设计 |
| 报告内容 | 5 | 说清楚：用了什么需求收集方法、怎么排优先级、怎么估算、迭代计划是什么 |
| 证据充分性 | 5 | 访谈记录/问卷截图/用户反馈，支撑报告中的每一个决策 |

### 个人系数（乘数，最高1）
- GitHub 有规律的 commit：0.5
- 助教观察到的参与度：0.5

**策略：第一次评估的分全在文档和原型，代码还不用写。重点是 Backlog 质量和报告逻辑。**

## 总结：100分策略

### 文档 > 代码
评委更看重"为什么这么做"，而不是代码写得多复杂。每个决策都要有记录。

### GitHub 是证据链
每个人的贡献必须在 GitHub 上可见。负责人要监控每个人的 commit 情况。

### 三次 Viva 都要准备
每次都会被问到"为什么"的问题。团队要统一口径，知道每个模块的设计理由。

### 核心功能做扎实，不要贪多
宁可 5 个功能做完美，也不要 15 个功能都半残。
